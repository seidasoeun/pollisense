# Demo Runbook

Use this sequence on demo day for the two-VM OpenNebula/k3s deployment.

## 1. Start VMs

Start VM1 first, then VM2.

```text
VM1: 172.16.100.2, k3s server/control-plane
VM2: 172.16.100.3, k3s worker/agent
```

## 2. Connect Through the Main Server

From Windows, keep the SOCKS tunnel open:

```bash
ssh -D 127.0.0.1:8888 -p 5033 <user>@<main-server>
```

From the main server, connect to VM1:

```bash
ssh root@172.16.100.2
cd ~/pollisense
```

Firefox on Windows should use SOCKS5 `127.0.0.1:8888`.

## 3. Recover After Reboot

On VM1:

```bash
bash scripts/opennebula-k3s/06-vm1-after-reboot.sh
```

This starts k3s, waits for PolliSense rollouts, and restarts the VM1 port-forward services when they are installed.

## 4. Verify k3s

```bash
kubectl get nodes -o wide
kubectl get storageclass
```

Expected: VM1 and VM2 are `Ready`. Use `kubectl` on VM1; VM2 normally has no kubeconfig.

## 5. Check Pods, PVC, and Services

```bash
kubectl get all -n pollisense -o wide
kubectl get pvc -n pollisense
kubectl get networkpolicy -n pollisense
```

Expected: PostgreSQL, backend, simulator, and frontend are running. The PVC is `Bound`.

## 6. Verify Port-Forward Services

```bash
systemctl status pollisense-frontend-portforward.service --no-pager
systemctl status pollisense-backend-portforward.service --no-pager
ss -lntp | grep -E ':(9999|8080)\b'
```

Expected listeners:

```text
172.16.100.2:9999
172.16.100.2:8080
```

## 7. Open the Dashboard Through SOCKS

Open this in Firefox on Windows:

```text
http://172.16.100.2:9999
```

Do not use `localhost:9999` from Windows. That points to the laptop, not VM1.

## 8. Show Records and Alerts

On VM1:

```bash
curl http://172.16.100.2:8080/actuator/health
curl http://172.16.100.2:8080/api/records
curl http://172.16.100.2:8080/api/alerts
curl http://172.16.100.2:8080/api/summary
kubectl logs -n pollisense deployment/pollisense-simulator --tail=50
```

Expected: health is up, records exist, alerts reflect low battery/connectivity/signal/module conditions generated by the simulator.

## 9. Scale the Backend

```bash
kubectl scale deployment pollisense-backend -n pollisense --replicas=3
kubectl rollout status deployment/pollisense-backend -n pollisense --timeout=180s
kubectl get pods -n pollisense -o wide -l app=pollisense-backend
curl http://172.16.100.2:8080/actuator/health
```

Return to one replica:

```bash
kubectl scale deployment pollisense-backend -n pollisense --replicas=1
kubectl rollout status deployment/pollisense-backend -n pollisense --timeout=180s
```

## 10. Persistence Test

```bash
curl http://172.16.100.2:8080/api/records
kubectl delete pod -n pollisense -l app=postgres --wait=false
kubectl rollout status deployment/postgres -n pollisense --timeout=180s
curl http://172.16.100.2:8080/api/records
```

Expected: records are still available after PostgreSQL restarts because data is stored on the PVC.

## 11. Collect Evidence

```bash
bash scripts/opennebula-k3s/08-vm1-collect-evidence.sh
```

Save the printed folder path for the report. Also capture:

- OpenNebula VM names, roles, CPU, RAM, disk, and IP addresses.
- Dashboard screenshot after records appear.
- `kubectl get nodes -o wide`
- `kubectl get all -n pollisense -o wide`
- `kubectl get pvc -n pollisense`
- `kubectl get networkpolicy -n pollisense`

## 12. Shutdown

Stop demo traffic if needed:

```bash
systemctl stop pollisense-frontend-portforward.service
systemctl stop pollisense-backend-portforward.service
```

Then shut down VM2 first and VM1 second from OpenNebula or the VM shell.
